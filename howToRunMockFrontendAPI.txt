# CrisisCloud (Flask + Leaflet + Supabase Postgres + NWS Weather API)

CrisisCloud is a Flask web application that serves a Leaflet-based frontend and pulls live weather, weekly forecast data, and active alerts from the National Weather Service (`weather.gov` / `api.weather.gov`) based on the current map center.

This version of the application uses a shared Supabase Postgres database for shelters, food banks, police stations, fire stations, hospitals, and towing services. Weather data is still retrieved live from the National Weather Service API.

## Current Features

- Live Leaflet map with custom resource icons
- Live weather, alerts, and weekly forecast from the National Weather Service
- Supabase-backed shared resource data
- Real-world locations for shelters, food banks, police, fire, hospitals, and towing services
- City filtering for:
  - All
  - Norfolk
  - Chesapeake
  - Portsmouth
  - Suffolk
  - Virginia Beach
- Resource filter panel for combining visible categories
- Organization login panel for updating resource status, phone number, and availability
- Emergency Contact Directory split into:
  - Police Stations
  - Fire Stations
  - Hospitals
  - Towing Services
- Separate Storm Shelter Directory and Food Bank Directory
- Collapsible directory sections
- Click-to-focus behavior that scrolls to the map and shows the selected location

## Install Dependencies

From a terminal in the project folder, run:

```powershell
pip install flask requests python-dotenv psycopg[binary]
```

## Environment Variables

Create a file named `.env` in the same directory as `crisisCloudBackend.py` with:

```env
NWS_USER_AGENT=CrisisCloud/1.0 (your-email@school.edu)
DATABASE_URL=postgresql://postgres.PROJECT_REF:YOUR_PASSWORD@aws-0-REGION.pooler.supabase.com:5432/postgres
```

Notes:

- `NWS_USER_AGENT` is required by the National Weather Service
- `DATABASE_URL` is the Supabase Postgres session pooler connection string
- do not place the `.env` file inside the `templates` folder

## Shared Database Setup

The app now uses a shared Supabase database instead of the local `crisiscloud.db` file.

That means:

- teammates do not need the same local `.db` file anymore
- all status changes save to the same shared database
- anyone using the same backend and same `.env` connection string will read and write the same data

If a teammate is connecting to the same Supabase project, make sure they have the same `DATABASE_URL` in their `.env`.

- `crisisCloudBackend.py`
- `.env`
- `newIcons/`
- `templates/crisisCloud.html`

They also need the `.env` file to point to the same Supabase project using the same `DATABASE_URL`.

## How Teammates Access The Shared Database

To use the same shared data as everyone else, teammates should:

1. Use the same codebase
2. Use the same `newIcons` folder
3. Use the same Supabase `DATABASE_URL` in `.env`
4. Run the Flask app or connect to the same deployed backend later

Because the database is now hosted in Supabase, status updates are no longer tied to one local SQLite file.

## Run Across Two Computers On The Same Network

If the Flask app is still running locally on one computer, teammates can still test together on the same network.

When you start the app, the terminal should show something like:

```text
Running on http://127.0.0.1:5000
Running on http://192.168.x.x:5000
```

To test on a second computer:

1. Keep the app running on the host computer.
2. Make sure both computers are on the same Wi-Fi or local network.
3. On the second computer, open the network address shown by Flask, for example:

```text
http://192.168.0.159:5000
```

Both computers will then connect to the same running backend, and the backend will read and write the same shared Supabase database.

## How Resource Data Works Now

- The app serves resource data through:
  - `GET /api/resources`
  - `POST /api/resources/<resource_id>`
- The frontend polls the backend so updates can appear after refresh or during the demo polling cycle
- Resource updates are now stored in Supabase Postgres instead of a local SQLite database

## How To Share The Latest Changes With The Team

The easiest way is to share:

- the latest code

- the correct `.env` format
- the shared Supabase project connection string

Teammates no longer need a shared copy of `crisiscloud.db` for live app data once they are using the Supabase-backed backend.

To add more real locations later:

1. Get the real address
2. Find the latitude and longitude
3. Insert or update the record in the Supabase `resources` table
4. Refresh the page or wait for polling

## Common Issues

### `TemplateNotFound`

Make sure `crisisCloud.html` is inside the `templates` folder and that the Flask route references:

```python
return render_template("crisisCloud.html")
```

### Icons Not Showing

- Confirm the `newIcons` folder exists in the project root
- Confirm the PNG files are inside `newIcons`
- Do not place `newIcons` inside `templates`
- Restart Flask after adding or moving icon files

### Weather Not Loading

- Confirm the frontend is calling `/api/weather/live`
- Confirm your `.env` file exists and includes `NWS_USER_AGENT`
- Check that the machine running Flask has internet access

### `.env` Not Loading

- Ensure the file is named exactly `.env`
- Keep it in the project root, not in `templates`
- Restart Flask after editing it

### Cannot Connect To Supabase

- Confirm `DATABASE_URL` is present in `.env`
- Confirm the password is correct
- If direct connection fails, use the Supabase session pooler connection string
- Confirm `psycopg[binary]` is installed

### Second Computer Cannot Connect

- Confirm both computers are on the same network if you are still hosting Flask locally
- Confirm the host computer is still running Flask
- Confirm Windows Firewall allows Python on private networks
- Use the host machine's network URL shown in the Flask terminal output
